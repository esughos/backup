## In openshift there is a concept of image stream and the logic behind this functionlity whenever we have images in any of our repositories there is a chance that those images can be updated and due to that the running application might be affected hence to mitigate this issue in openshift this image stream will not only locate the image in the repository but they point to the image-id in the repositroy which is always unique for a image file and hence there is a no chance to update the particular image.

## In Openshift there is a concept of Source2Image i.e. whenever you want to build a particular application you just mention the source code repository like github/gitlab/perforce/SCM it will build the application from the docker file and convert it into a docker image and then push the image into it's own repository. Other than this it is also possible to build image in a magic way just specify your source code only in git repo and openshift will build an application image from the source code (it will open your source code and see what are the things mentioned in the code then it'll create a docker file based on that and form a application docker image) and pushed it into openshift container registry. Beside these we can manually build a docker file in openshift by editing the yaml file for a particular application.

## In openshift every pod which is deployed in different worker and master node need to communicate with each other hence to have a communication within the pods SDN (Software/Service Defined Network) which is also called Overlay Network. The Overlay Network use Open VSwitch which have "VLAN tagging", "Trunking", "LACP", "Port Mirroring".

However, the Overlay Network should have an IP pool like 10.128.0.0/14 and if there are 3 Pods then all the 3 Pods should have subnet pool from the Overlay Network pool like 1st Pod subnet should have 10.128.0.0/23, 2nd Pod subnet should have 10.128.2.0/23 and 3rd Pod subnet should have 10.128.4.0/23. Now, 1st Pod should receive an IP from the 1st Pod subnet pool and likewise for others.

Now each Pod should communicate with another Pod by it's name so openshift uses SkyDNS as a DNS service to resolve hostname into Ip-Address as Pod should not directly communicate with IP-Address as the process is not secure if the Pod restarted then the IP-Address of the Pod get change and hence outage may happen.

Each project in openshift within each other communicate with OVS Multitenant with a unique virtual IP and that can be configured through service.

Now to communicate to the external world service works with route to expose application to external users. Now, route can be attched with "LoadBalancer" to serve this purpose. There are three load balancer present "Source" which has a direct relationship with the "Pod" and the "External users" i.e. it acts as a sticky session, "RoundRobin" routes traffic randomly based on the load of the host and "leastconn" routes towards endpoints with lowest number of connections.
